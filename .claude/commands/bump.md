description:
  Bump the CHANGELOG `[Unreleased]` section to a concrete version and stage the
  commit. Usage — `/bump [patch|minor|major]`, e.g. `/bump minor`. $ARGUMENTS
---

You are about to roll `CHANGELOG.md` for a release. Follow these steps:

1. Decide the next version. If `$ARGUMENTS` is `patch`, `minor`, or `major`,
   bump the last tag accordingly. Otherwise ask the user. Use
   `git describe --tags --abbrev=0` to find the current tag. Rule reminder: this
   repo follows [`.claude/skills/release-bump.md`](../skills/release-bump.md).
   When in doubt, patch.
2. Read `CHANGELOG.md`. Rename its `[Unreleased]` section header to
   `[X.Y.Z] - YYYY-MM-DD` (use today's date). Insert a fresh empty
   `[Unreleased]` section above it with the empty subsection headers
   (`### Added`, `### Changed`, `### Fixed`, `### Deprecated`, `### Contracts`,
   `### Security`).
3. Do NOT touch `src/nyc311/_version.py` — it's generated by `hatch-vcs` from
   the git tag. If a hand-edited `_version.py` appears, stop and ask.
4. Stage the change with `git add CHANGELOG.md`. Do NOT commit; leave staged so
   the user can review and commit by hand (or sign).
5. Print a short summary:
   - The chosen version.
   - The bullets that rolled out of `[Unreleased]` into `[X.Y.Z]`.
   - The next step for the user:
     `git commit -m "chore(release): bump to vX.Y.Z"` then
     `git tag vX.Y.Z && git push --follow-tags`.

Do not push, do not tag, do not sign. The user owns the tag.
